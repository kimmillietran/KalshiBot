# PR-12.1C — Official Kalshi WebSocket Capture Spike

## Summary

Adds a minimal, safe capture spike proving KalshiBot can subscribe to live KXBTC15M orderbook updates, reconstruct top-of-book state, optionally capture BTC spot, and write auditable JSONL artifacts plus a health summary.

## Why this exists

Post-M12.0 analysis showed historical KXBTC15M data is close-only and unusable for executable microstructure research. Forward capture must prove live orderbook plumbing before building a full MVP.

## Capture assumptions

- **Top-of-book:** derived from YES/NO bid ladders; asks computed as `100 - opposite best bid`.
- **Transport:** Kalshi trade API v2 WebSocket (`orderbook_delta` channel).
- **Persistence:** append-only raw JSONL first; normalization deferred.
- **Safety:** capture only — no orders, no strategy decisions, no trading clients imported.

## Limitations

- Spike runs for a fixed duration; not a daemon or production capture service.
- WS auth signing is interface-ready but credentials are env-based.
- Ask sizes use opposite-side bid size conservatively when deriving complements.
- Sequence gaps mark the book invalid until a fresh snapshot arrives.
- Queue position and adverse selection are not modeled.

## CLI

```bash
npm run live:kalshi-ws-capture-spike -- \
  --series KXBTC15M \
  --duration-seconds 300 \
  --max-markets 1 \
  --output-dir data/live-capture/kalshi-ws-spike \
  --dry-run
```

## Outputs

Per run (`<runId>`):

- `data/live-capture/kalshi-ws-spike/<runId>/raw-messages.jsonl`
- `data/live-capture/kalshi-ws-spike/<runId>/top-of-book.jsonl`
- `data/live-capture/kalshi-ws-spike/<runId>/btc-spot.jsonl` (optional)
- `data/live-capture/kalshi-ws-spike/<runId>/capture-health.json`
- `data/reports/kalshi-ws-capture-spike.html`
