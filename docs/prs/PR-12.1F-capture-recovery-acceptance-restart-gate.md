# PR 12.1F — Capture Recovery Acceptance Harness and Eight-Hour Restart Gate

Proves the capture stack is ready before another eight-hour Hypothesis #3
run is started. Two complementary gates:

1. A **deterministic recovery acceptance harness** that drives the full
   production capture orchestrator through a scripted WebSocket transport
   and verifies the entire gap-recovery lifecycle from real artifacts.
2. A **short live capture smoke gate** (15–30 minutes) on the canonical
   authenticated capture path, followed by an explicit audit of the exact
   run and a frozen restart acceptance decision.

This milestone changes no trading or hypothesis thresholds.

## Part A — Deterministic recovery acceptance

```bash
npm run research:capture-recovery-acceptance
```

Runs entirely in memory (ephemeral RSA credentials, scripted transport,
in-memory IO) through `runForwardQuoteCapture` and replays this exact
scenario:

1. Successful authenticated connection.
2. Subscribe command; server acknowledgement with a server `sid`.
3. Valid initial order-book snapshot (seq 1) and normal deltas (seq 2, 3).
4. One intentional sequence discontinuity (seq 10).
5. Exactly one `update_subscription get_snapshot` recovery request using the
   acknowledged `sid`.
6. Additional deltas (seq 11, 12) arriving while recovery is pending —
   quarantined, never applied.
7. Recovery `ok` acknowledgement, fresh snapshot (seq 100), and accepted
   post-recovery deltas (seq 101, 102).
8. Market unsubscribe with acknowledgement (rollover path).
9. Graceful finalization, full writer drain, atomic health publication, and
   a terminal `completed` run status.

The command prints one machine-readable acceptance report (observed
metrics, per-check results, wire transcript) and exits nonzero when any
acceptance requirement fails, including: more than one gap episode or
recovery request, quarantined deltas leaking into applied records, a
runaway `sequenceGapCount`, missing unsubscribe acknowledgement, undrained
persistence streams, a non-`completed` terminal status, non-atomic health
publication, or credential material in any artifact.

## Part B — Short live capture smoke gate

```powershell
./run-capture-restart-smoke.ps1                    # 20-minute smoke
./run-capture-restart-smoke.ps1 -DurationMinutes 15
```

The wrapper (never an eight-hour capture; duration is validated to
15–30 minutes):

1. **Refuses to run when another capture is active.** Preflight:
   `npm run research:capture-restart-gate -- --assert-no-active-capture`
   fails when any run directory carries a strictly valid
   `active`/`finalizing` status.
2. **Runs the canonical capture path** — authenticated Kalshi WebSocket
   capture with BTC spot and the same top-of-book throttle intended for
   eight-hour runs (`-TopOfBookThrottleMs`, default 0 = the capture CLI
   default).
3. **Identifies the exact run directory it created** from the capture CLI's
   machine-readable output (`runId` + `outputDir`). It never selects
   "latest". The capture process exits only after producers quiesce, the
   buffered writer fully drains, and the terminal status publishes — so
   process exit is terminal writer completion.
4. **Audits that exact run**: `research:capture-health-audit`,
   `research:bid-size-coverage-audit`, and
   `research:capture-health-reconciliation`, all by `--capture-run-dir`.
5. **Evaluates the restart gate** and prints one machine-readable readiness
   summary. The wrapper's exit code is the gate's exit code.

### Restart acceptance criteria (frozen)

`restartEightHourCaptures: true` only when all of the following hold:

- terminal run status is `completed` (strictly validated, identity-matched);
- audit verdict is `capture-research-ready` (schema-validated, run-scoped);
- run duration is within tolerance of the requested smoke duration
  (±10%, floor 60s);
- BTC join coverage ≥ 0.90 and valid-book share ≥ 0.90;
- no terminal WebSocket failure and a `duration-complete` end reason;
- no unresolved recovery (outstanding recoveries, timeouts, exhausted
  retries, pending commands);
- no command-level WebSocket errors or acknowledgement timeouts;
- every gap episode has a recovery outcome;
- suppressed resyncing deltas are reported separately and
  `sequenceGapCount` equals the distinct episode count;
- all persistence streams drained with no writer failure;
- no stale or partial health/audit artifact was used (source-artifact
  fingerprints positively re-verified).

A real sequence gap is **not** required to occur during the live smoke; the
deterministic harness (Part A) proves the recovery path.

### Operator summary

The gate prints a single stable-ordered JSON summary
(`schemaVersion: 1`): `runId`, `runDir`, `durationSeconds`,
`topOfBookCount`, `btcSpotCount`, `validBookShare`,
`btcJoinCoverageShare`, `gapEpisodeCount`, `recoveryRequestCount`,
`recoverySuccessCount`, `recoveryFailureCount`,
`suppressedWhileResyncingCount`, `writerBackpressureCount`,
`allStreamsDrained`, `nativeHealthStatus`, `runStatusState`,
`auditVerdict`, `auditFingerprintsVerified`,
`restartEightHourCaptures`, `failureReasons`.

`parseCaptureRestartGateSummary` provides strict fail-closed validation for
automation consuming the summary.

## Module layout

- `src/lib/data/live/forwardQuoteCapture/captureRecoveryAcceptance/` —
  scripted transport, in-memory harness, pure acceptance evaluator.
- `src/lib/data/live/forwardQuoteCapture/captureRestartGate/` — pure restart
  gate evaluator, strict summary schema, active-capture detection.
- `scripts/research/runCaptureRecoveryAcceptance.ts` — Part A CLI.
- `scripts/research/evaluateCaptureRestartGate.ts` — preflight + gate CLI.
- `run-capture-restart-smoke.ps1` — Part B operator wrapper.

## Artifact hygiene

The deterministic harness writes nothing to disk. The live smoke writes
normal capture artifacts under `data/live-capture/forward-quotes/<runId>/`
(gitignored); generated artifacts are never committed.
